# 미냥이 지갑 (KITTYWALLET) AI 작업 규칙

## 프로젝트 개요

미냥이 지갑은 PWA 기반 모바일 가계부 서비스이다.

본 프로젝트는 문서 중심 개발(Document Driven Development)을 원칙으로 하며, 모든 구현은 정의된 문서를 기준으로 수행한다.

# 최우선 원칙

문서 > 코드 > 추정

정의되지 않은 기능은 임의 구현하지 않는다.

정책이 없는 경우 반드시 문서를 확인하거나 문서 작성 필요성을 먼저 제안한다.

작업 시작 전 반드시 `docs/00_project/문서인덱스.md`를 확인하여 관련 문서를 식별한다.

# 작업 시작 순서

작업 전 문서 참조 순서는 `docs/00_project/문서인덱스.md`의 최상위 기준과 작업 유형별 빠른 참조 경로를 따른다.

본 문서는 최상위 AI 작업 규칙이며, 상세 문서 탐색 순서는 문서인덱스에서 관리한다.

## GitHub Issue 기반 개발 절차

GitHub Issue를 처리하는 작업은 항상 다음 순서를 따른다.

1. GitHub에서 대상 Issue 본문, 상태, 라벨, 댓글, 서브이슈, 선행/의존 Issue 상태를 확인한다.
2. Issue 본문의 작업 범위, 완료 조건, 서브이슈 체크박스를 작업 시작 전에 확인한다.
3. 선행 Issue가 open 상태이거나 완료되지 않은 경우 구현을 시작하지 않고 작업을 종료한 뒤 사용자에게 알려준다.
4. 선행 Issue 때문에 작업을 중단하는 경우 Project Status는 임의 변경하지 않는다.
5. 작업을 실제로 시작하는 경우 GitHub Project 등록 여부를 확인하고, 미등록이면 Project에 먼저 추가한 뒤 Project Status를 `In Progress`로 변경한다.
6. 서브이슈가 있으면 서브이슈를 먼저 검토한다.
7. 서브이슈가 개발 범위이고 현재 작업에서 함께 처리 가능하면 서브이슈까지 구현, 검증, PR 연결 대상에 포함한다.
8. 서브이슈가 독립적으로 먼저 완료되어야 하는 범위이면 해당 서브이슈 작업을 우선 진행하고 상위 Issue 작업은 보류한다.
9. 개발 시작 전 반드시 새 작업 브랜치를 생성한다.
10. 로컬에서 구현한 뒤 lint, type-check, test, build, 필요한 Docker/health 검증까지 성공해야 커밋한다.
11. 검증 성공 후에만 커밋, push, PR 생성을 진행한다.
12. PR 본문에는 상위 Issue와 처리한 서브이슈를 모두 연결한다.
13. PR 생성 후 merge 전까지 Project Status는 `In Progress`로 유지한다.
14. PR merge 또는 Issue close 완료 후 Project Status를 `Done`으로 변경한다.
15. Issue 완료 조건에 검사 실패 시 merge 금지 또는 필수 CI check가 포함된 경우 GitHub 브랜치 보호 규칙과 필수 상태 체크 설정을 확인한다.
16. 작업자가 완료한 작업 범위, 완료 조건, 서브이슈 체크박스는 실제 구현, 테스트, 검증, PR 연결이 거짓 없이 완료된 항목만 GitHub Issue 본문에서 갱신한다.

Issue 완료 여부, 선행 관계, 서브이슈 상태는 로컬 추정이나 문서가 아니라 항상 GitHub Issue 상태를 기준으로 판단한다.

각 단계로 넘어가기 전에는 해당 단계와 관련된 문서에 변경 사항이 있는지, 추가로 지켜야 할 기준이 있는지 확인한다. 개발, 테스트, Docker, Git, PR, 이슈 갱신 등 모든 단계는 관련 문서 확인 후 진행한다.

# GitHub Issue 기반 작업 규칙

GitHub Issue에 메인이슈와 서브이슈가 함께 있는 경우, AI는 작업 시작 전에 메인이슈와 서브이슈의 범위, 완료 조건, 선행 관계를 확인한다.

메인이슈와 서브이슈 작업은 가능한 한 커밋을 분리한다.

서브이슈는 각각 개별 작업 브랜치를 가져야 하며, 개별적으로 개발이 완료되면 해당 서브이슈 단위로 커밋하고 PR을 생성한다.

서브이슈 PR을 생성한 뒤에는 해당 서브이슈의 Project Status를 `In Progress`로 변경하고, PR merge 또는 Issue close 완료 후 `Done`으로 변경한다.

서브이슈별 커밋 또는 PR 분리가 불가능한 경우, AI는 작업을 임의로 묶지 않고 사용자에게 불가능한 이유를 먼저 설명한다.

불가 사유에는 다음 내용을 포함한다.

* 서브이슈 간 코드 변경이 기술적으로 분리하기 어려운 이유
* 개별 PR 생성 시 빌드, 테스트, 의존성, 배포 흐름에 문제가 생기는 이유
* 하나의 PR로 묶어야 한다면 포함되는 메인이슈와 서브이슈 목록

연관 있는 이슈들이 같은 변경 범위와 검증 흐름을 공유하는 경우에만 여러 이슈를 하나의 PR로 묶어 처리할 수 있다.

여러 서브이슈를 하나의 PR로 처리하는 경우에도 PR 본문에는 포함된 메인이슈와 모든 서브이슈 번호를 항상 명시하여 GitHub Issue 상태 연동이 가능하도록 한다.

서브이슈들끼리만 하나의 PR로 묶는 경우에도 PR 본문에는 묶여 있는 모든 서브이슈 번호를 항상 기재하여 GitHub Issue 상태 연동이 가능하도록 한다.
