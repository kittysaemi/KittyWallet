# 개발목표

## 1. 문서 목적

본 문서는 미냥이 지갑 프로젝트의 개발 방향과 핵심 목표를 정의한다.

개발 목표는 단순 기능 구현이 아니라 다음 기준을 충족하는 것을 목적으로 한다.

* 유지보수 가능한 구조
* 확장 가능한 구조
* 정책 기반 개발
* 모바일 중심 사용자 경험
* 오프라인 대응 가능한 PWA 구조
* 프론트/백엔드 분리 구조
* AI 협업 가능한 개발 구조

---

# 2. 프로젝트 개발 목표

## 2.1 모바일 중심 가계부 서비스 제공

미냥이 지갑은 모바일 환경에서 빠르게 수입 및 지출 내역을 기록하고 확인할 수 있는 모바일 중심 가계부 서비스를 목표로 한다.

주요 목표:

* 빠른 거래 입력
* 직관적인 사용 흐름
* 최소 입력 기반 UI
* 모바일 터치 중심 인터페이스
* 하단 메뉴 기반 이동 구조

---

## 2.2 PWA 기반 모바일 앱 경험 제공

웹 기반 서비스이지만 모바일 앱과 유사한 사용자 경험 제공을 목표로 한다.

적용 목표:

* 홈 화면 설치 지원
* 전체화면 앱 형태 지원
* Service Worker 기반 캐시 처리
* 오프라인 상태 대응
* 빠른 초기 로딩
* 모바일 Safe Area 대응

---

## 2.3 유지보수 가능한 구조 설계

기능 추가 및 수정 시 전체 코드 영향도를 최소화할 수 있는 구조를 목표로 한다.

구조 목표:

* 기능 기반 폴더 구조
* 공통 로직 분리
* API 추상화 구조
* 상태 관리 분리
* UI와 비즈니스 로직 분리
* Repository 패턴 적용

---

## 2.4 API 변경 대응 가능한 구조 설계

백엔드 API 변경 또는 추가 시 프론트엔드 전체 수정이 발생하지 않는 구조를 목표로 한다.

적용 방향:

* API Client 추상화
* Repository Interface 구조
* DTO 기반 데이터 처리
* 공통 Response Wrapper 사용
* Feature 단위 API 분리

---

## 2.5 프론트엔드 / 백엔드 분리 구조 구성

프론트엔드와 백엔드를 독립적으로 유지보수 가능한 구조로 설계한다.

목표:

* 독립 배포 가능 구조
* 공통 타입 공유 구조
* API 명세 기반 개발
* 상태 기반 데이터 처리

---

## 2.6 정책 기반 개발 구조 구성

모든 기능은 정책 정의를 기준으로 개발하는 것을 목표로 한다.

적용 기준:

* 상태 정의 기반 UI 처리
* 예외 처리 기준 문서화
* API 오류 처리 규칙 통일
* 데이터 무결성 유지
* Soft Delete 우선 적용

---

## 2.7 오프라인 및 동기화 지원

네트워크가 불안정한 환경에서도 기본 기능 사용이 가능하도록 설계한다.

목표:

* 최근 데이터 캐시
* 오프라인 거래 저장
* 재연결 시 자동 동기화
* 충돌 최소화 구조
* 상태 기반 동기화 처리

---

## 2.8 통계 및 데이터 분석 지원

거래 데이터를 기반으로 소비 흐름과 월간 사용 현황을 확인할 수 있도록 한다.

지원 목표:

* 월별 수입/지출 통계
* 카테고리별 소비 분석
* 최근 거래 조회
* 카드 사용 분석
* 기간별 검색 및 집계

---

# 3. 개발 구조 목표

## 3.1 기능 중심 구조 적용

화면 중심이 아닌 기능 중심 구조를 적용한다.

예시:

* transaction
* account
* category
* statistics

기준으로 기능을 분리한다.

---

## 3.2 공통 모듈 구조 적용

재사용 가능한 공통 모듈을 별도 관리한다.

대상:

* API Client
* 공통 UI
* 상태 관리
* 유틸 함수
* 타입 정의

---

## 3.3 상태 기반 UI 처리

모든 화면은 상태 기반으로 동작하도록 구성한다.

필수 상태:

* loading
* empty
* error
* offline
* success

---

## 3.4 공통 예외 처리 적용

서비스 전반에 동일한 오류 처리 기준을 적용한다.

대상:

* API 실패
* 네트워크 오류
* 인증 만료
* 저장 실패
* 동기화 실패

---

# 4. 데이터 구조 목표

## 4.1 거래 중심 데이터 구조 적용

수입, 지출, 카드 사용 내역을 하나의 거래 구조에서 통합 관리한다.

기준:

* transaction_type
* wallet_type

---

## 4.2 사용자 기준 데이터 분리

모든 데이터는 user_id 기준으로 사용자별 분리 관리한다.

---

## 4.3 Soft Delete 우선 적용

데이터 무결성과 통계 보존을 위해 물리 삭제보다 논리 삭제를 우선 적용한다.

---

## 5.1 테스트 가능한 구조 유지

테스트 가능한 구조를 유지하여 기능 안정성을 확보한다.

목표:

Codex 및 AI 기반 개발 도구가 문서 기준으로 안정적으로 작업 가능한 구조를 유지한다.

기준:

* 문서 우선 개발
