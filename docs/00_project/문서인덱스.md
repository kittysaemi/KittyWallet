# 문서인덱스

## 1. 문서 목적

본 문서는 미냥이 지갑 프로젝트의 전체 문서 구조 및 참조 우선순위를 정의한다.

모든 개발자와 AI(Codex, Claude Code, Copilot)는 작업 시작 전 본 문서를 기준으로 필요한 문서를 탐색한다.

---

## 2. 문서 참조 원칙

작업자는 구현 전에 다음 최상위 기준을 따른다.

1. `AGENTS.md`
2. `docs/00_project/문서인덱스.md`
3. 본 문서의 작업 유형별 필수 문서
4. 관련 정책/API/ERD/화면/상태 문서

작업 유형별 필수 문서는 아래 빠른 참조 경로를 기준으로 확인한다.

1. 프로젝트 기준 문서
2. 작업 영역의 정책 문서
3. 화면/상태/API/데이터 문서
4. 프론트엔드 또는 백엔드 아키텍처 문서
5. PWA, 디자인 시스템, 테스트, 배포 문서

문서 간 내용이 충돌할 경우 다음 우선순위를 따른다.

| 우선순위 | 문서 범위 | 판단 기준 |
|---|---|---|
| 1 | 정책 문서 | 기능 허용 여부, 금지 사항, 비즈니스 규칙 |
| 2 | API 문서 | 프론트엔드와 백엔드 간 계약 |
| 3 | 데이터 문서 | DB 구조, 관계, 무결성 기준 |
| 4 | 화면/상태 문서 | 사용자 화면, UI 상태, 예외 처리 |
| 5 | 아키텍처 문서 | 구현 구조와 코드 배치 |
| 6 | 테스트/배포 문서 | 검증, 운영, 릴리즈 기준 |

충돌이 발견되면 임의 구현하지 않고 관련 문서를 함께 수정하거나 이슈로 남긴다.

## 6. AI 작업 규칙

AI는 작업 시작 시 다음 규칙을 따른다.

1. 사용자 요청과 관련된 문서를 본 인덱스에서 먼저 찾는다.
2. 단일 문서만 보고 구현하지 않는다.
3. 정책, API, ERD, 화면 문서가 충돌하면 충돌 내용을 먼저 보고한다.
4. 누락된 API나 상태 정의가 있으면 임의로 확정하지 않고 문서 보완 필요 항목으로 남긴다.
5. PWA, 오프라인, 동기화 작업은 반드시 정책 문서와 PWA 문서를 함께 확인한다.
6. 테스트 또는 배포 작업은 `11_testing`, `12_deployment` 문서를 함께 확인한다.

## 7. 문서 보완 시 기준

새 문서를 추가하거나 기존 문서를 수정할 때는 다음 기준을 따른다.

문서 본문은 최신 기준만 유지한다.

과거 기준을 남겨야 할 경우 본문에 함께 두지 않고 `[DEPRECATED]` 섹션 또는 변경 이력으로 분리한다.

| 상황 | 처리 |
|---|---|
| 신규 기능 추가 | 정책, 화면, API, 데이터, 테스트 문서 반영 |
| API 변경 | API 문서, 화면 문서, PWA 캐시 문서, 테스트 문서 반영 |
| DB 변경 | ERD, 데이터구조정의, Repository 정책, API 문서 반영 |
| 상태 추가 | 상태정의, 화면정의, PWA상태정의 또는 상태관리 문서 반영 |
| 오프라인 동작 변경 | 오프라인정책, 동기화정책, PWA 문서, 테스트 문서 반영 |
| 배포 정책 변경 | 배포정책, 배포절차, 운영체크리스트, CI/CD 문서 반영 |

## 8. 최종 기준

본 문서는 미냥이 지갑 프로젝트 문서 탐색의 시작점이다.

개발자는 작업 전 본 문서를 통해 관련 문서를 찾고, 문서 간 기준이 맞는지 확인한 뒤 구현을 진행한다.

AI는 본 문서를 기준으로 필요한 문서를 순차적으로 참조하며, 문서 충돌이나 누락이 발견되면 구현보다 먼저 사용자에게 명확히 보고한다.
