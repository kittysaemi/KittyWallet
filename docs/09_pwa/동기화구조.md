# 동기화구조

## 1. 문서 목적

본 문서는 미냥이 지갑 PWA의 오프라인 데이터 동기화 구조와 처리 기준을 정의한다.

동기화 기능은 오프라인 상태에서 저장된 데이터를 네트워크 복구 후 서버와 일관성 있게 동기화하기 위해 사용한다.

본 문서는 다음 문서와 연계된다.

* PWA개요
* ServiceWorker정의
* 캐시정책
* 오프라인동작정의
* PWA상태정의
* 거래정책
* 동기화정책

---

# 2. 동기화 개요

미냥이 지갑은 오프라인 상태에서도 거래를 저장할 수 있다.

오프라인 저장 데이터는 서버에 즉시 반영되지 않는다.

따라서 별도의 Sync Queue를 통해 서버와 동기화를 수행한다.

---

동기화 구조

```text
거래 저장
 ↓
IndexedDB 저장
 ↓
Sync Queue 등록
 ↓
Network 복구
 ↓
동기화 실행
 ↓
서버 반영
 ↓
Queue 제거
```

---

# 3. 동기화 원칙

## 원칙 1

사용자 데이터는 절대 유실되면 안 된다.

---

## 원칙 2

동일 데이터는 중복 저장되면 안 된다.

서버 중복 방지는 `(user_id, sync_client_id, client_temp_id)` 기준으로 처리한다.

---

## 원칙 3

동기화 실패 시 재시도 가능해야 한다.

---

## 원칙 4

동기화 중 앱 종료가 발생해도 복구 가능해야 한다.

---

## 원칙 5

최종 데이터 기준은 서버이다.

---

```text

```

---

이유

```text

```

---

## 서버 저장 완료 거래 삭제

```text
DELETE Queue 생성
```

---

예시

```json
{
  "server_id": "501",
  "sync_action": "DELETE"
}
```

---

## 삭제 우선순위

| 상황 | 처리 |
|---|---|
| CREATE 대기 중 삭제 | CREATE Queue 제거 |
| UPDATE 대기 중 삭제 | UPDATE Queue 제거 후 DELETE Queue 생성 |
| DELETE 대기 중 수정 | UPDATE Queue 생성 금지 |
| DELETE 중복 요청 | 기존 DELETE Queue 유지 |

삭제는 수정보다 우선한다.

서버에 존재하지 않는 미동기화 거래는 동기화 대상에서 제거한다.


```text

```

# 12. 충돌 처리 정책

충돌은 다음 상황을 의미한다.

```text
동일 거래
 ↓
서버 수정
 ↓
클라이언트 수정
```

---

처리 기준

```text
MVP 기준: 클라이언트 우선 자동 반영
사용자 선택 UI 없음
```

---

정책 상세

| 상황 | 처리 |
|---|---|
| payload.updated_at이 서버 updated_at보다 최신 | 클라이언트 데이터를 서버에 반영 (정상 처리) |
| payload.updated_at이 서버 updated_at보다 과거 | 클라이언트 데이터를 서버에 강제 반영 (클라이언트 우선). SYNC_HISTORY에 CONFLICT 기록 |

서버는 CONFLICT를 `SYNC_HISTORY`에 기록하지만 응답 `sync_result`는 처리 완료로 반환한다.

---

예시

```text
Server updated_at: 13:00
Client updated_at: 13:10
```

결과

```text
Client 우선 반영 (정상 케이스)
```

```text
Server updated_at: 13:10
Client updated_at: 13:00
```

결과

```text
Client 강제 반영 + SYNC_HISTORY CONFLICT 기록
```

---

# 13. 서버 저장 구조

| 서버 테이블 | 역할 |
|---|---|
| TRANSACTION | 거래 원장, client_temp_id/sync_client_id/synced_at 저장 |
| SYNC_CLIENT | 클라이언트 또는 기기 식별, last_synced_at 저장 |
| SYNC_HISTORY | 항목별 서버 반영 성공/실패/충돌 이력 저장 |

동기화 성공 시:

```text
TRANSACTION.synced_at 갱신
SYNC_CLIENT.last_synced_at 갱신
SYNC_HISTORY SUCCESS 기록
```

충돌 발생 시:

```text
SYNC_HISTORY CONFLICT 기록
클라이언트 데이터를 서버에 강제 반영(클라이언트 우선)
응답은 처리 완료로 반환
사용자 선택 UI 없음
```

실패 시:

```text
SYNC_HISTORY FAILED 기록
Queue 유지
재시도 정책에 따라 자동 재시도
```

# 16. 계좌 잔액 반영

거래 저장 성공

```text
Transaction 저장
 ↓
잔액 반영
```

---

동기화 성공

```text
서버 재계산
```

계좌 거래 동기화 시 서버는 계좌의 `initial_balance`와 삭제되지 않은 계좌 거래를 기준으로 날짜별 잔액 흐름을 최종 검증한다. 마이너스 미허용 계좌는 어느 날짜의 잔액도 0 미만이 될 수 없고, 마이너스 허용 계좌는 `-negative_balance_limit`보다 작아질 수 없다.

검증 실패 시 해당 Queue 항목은 실패 처리하며 `ACCOUNT_004` 오류를 기록한다. 클라이언트는 재시도만 반복하지 않고 사용자가 금액, 날짜, 계좌를 수정할 수 있는 실패 상태를 표시한다.

# 27. 카드할부 동기화 정책

## 할부 생성 Queue payload

오프라인 상태에서 카드할부 등록 시 다음 payload를 Sync Queue에 저장한다.

```json
{
  "operation": "CREATE",
  "entity": "installment",
  "client_temp_id": "uuid-xxx",
  "payload": {
    "transaction_type": "EXPENSE",
    "wallet_type": "CARD",
    "wallet_id": 1,
    "category_id": 2,
    "amount": 120000,
    "transaction_date": "2026-06-20",
    "memo": "냉장고 구매",
    "installment": {
      "installment_months": 3
    }
  }
}
```

## 중복 생성 방지

| 항목 | 정책 |
|---|---|
| 식별 기준 | client_temp_id 기준으로 서버 중복 생성 방지 |
| 동기화 성공 | 부모 할부와 월별 거래 모두 생성된 경우만 성공 처리 |
| 동기화 실패 | Queue 재시도. 부분 생성 방지를 위해 트랜잭션으로 처리 |
| 오프라인 수정 | 동기화 전 할부 거래 수정은 Queue payload 갱신으로 처리 |
