# 데이터 구조 정의

## 1. 문서 목적

본 문서는 미냥이 지갑 서비스의 데이터 저장 구조와 관계를 정의한다.

데이터 구조 정의는 다음 목적을 가진다.

* 프론트/백엔드 공통 데이터 기준 제공
* API 응답 구조 기준 제공
* 통계 및 검색 기준 정의
* 상태 관리 및 오프라인 동기화 기준 정의
* 유지보수 및 기능 확장 기준 제공

본 문서는 단순 ERD 문서가 아니라 실제 개발 및 운영 기준 문서로 사용한다.

---

# 2. 데이터 설계 원칙

## 2.1 사용자 데이터 분리

모든 데이터는 user_id 기준으로 사용자별 분리 저장한다.

대상:

* 거래내역
* 계좌
* 카드
* 카테고리
* 아이콘
* 사용자 설정
* 동기화 클라이언트
* 동기화 이력

---

## 2.2 Soft Delete 우선 정책

데이터 보존 및 통계 무결성을 위해 물리 삭제보다 Soft Delete를 우선 사용한다.

적용 기준:

* TRANSACTION → deleted_yn
* CATEGORY → show
* ICON → show
* ACCOUNT/CARD → use_yn

---

## 2.3 거래 중심 구조

수입/지출 거래 내역은 TRANSACTION 테이블에서 통합 관리한다.

구분 기준:

* transaction_type: 수입/지출
* wallet_type: 계좌/카드 결제수단

거래의 `wallet_id`는 `wallet_type`에 따라 다음 값을 참조한다.

| wallet_type | wallet_id 참조 대상 |
| --- | --- |
| ACCOUNT | ACCOUNT.account_id |
| CARD | CARD.card_id |

---

## 2.4 오프라인 동기화 고려

PWA 오프라인 저장 및 동기화를 위해 다음 기준을 적용한다.

* created_at
* updated_at
* deleted_yn
* client_temp_id
* sync_client_id
* synced_at
* last_synced_at

컬럼을 모든 주요 테이블에 공통 적용 가능하도록 설계한다.

MVP 동기화 대상은 거래만 포함한다. 계좌, 카드, 카테고리, 아이콘, 사용자 설정은 온라인 API 기준으로 저장한다.

---

# 3. 공통 컬럼 규칙

| 컬럼명        | 설명       |
| ---------- | -------- |
| created_at | 생성일시     |
| updated_at | 수정일시     |
| deleted_yn | 삭제 여부    |
| use_yn     | 사용 여부    |
| show       | 화면 표시 여부 |
| client_temp_id | 클라이언트 임시 ID |
| sync_client_id | 동기화 클라이언트 ID |
| synced_at | 서버 반영 완료 일시 |
| last_synced_at | 마지막 동기화 일시 |

---

# 4. 공통 데이터 타입 규칙

| 타입  | 기준            |
| --- | ------------- |
| ID  | BIGINT        |
| 금액  | DECIMAL(15,0) |
| 상태값 | BOOLEAN 또는 ENUM 문자열 |

## transaction_type

| 값       | 의미    |
| ------- | ----- |
| INCOME  | 수입    |
| EXPENSE | 지출    |

---

## wallet_type

| 값       | 의미 |
| ------- | -- |
| ACCOUNT | 계좌 |
| CARD    | 카드 |

---

## 거래 유형과 지갑 유형 조합

| transaction_type | wallet_type | 허용 여부 | 설명 |
|---|---|---|---|
| INCOME | ACCOUNT | Y | 계좌 수입 거래 |
| EXPENSE | ACCOUNT | Y | 계좌 지출 거래 |
| EXPENSE | CARD | Y | 카드 지출 거래 |
| INCOME | CARD | N | 카드는 수입 거래를 지원하지 않음 |

# 6. USER 테이블

목적:
사용자 인증 및 사용자 기본 정보 관리

주요 역할:

* 로그인
* 사용자 식별
* 데이터 소유 기준

---

# 7. TRANSACTION 테이블

목적:
수입/지출 거래 내역 통합 관리

핵심 기준:

* 거래 중심 구조
* transaction_type은 INCOME/EXPENSE만 사용
* wallet_type은 ACCOUNT/CARD 결제수단을 의미
* client_temp_id는 오프라인 생성 거래의 중복 방지 키
* sync_client_id는 거래를 업로드한 클라이언트 또는 기기를 식별
* synced_at은 서버 반영 완료 시점을 기록
* 통계 기준 테이블
* 검색 기준 테이블

주요 정책:

* 미래 날짜 저장 불가
* 금액 0 이하 저장 불가
* `transaction_type=INCOME`, `wallet_type=CARD` 조합 저장 불가
* `(user_id, sync_client_id, client_temp_id)` 기준 중복 저장 방지
* 동기화 완료 시 synced_at 갱신
* 삭제 시 deleted_yn=true
* 수정 시 잔액 재계산

---

# 8. ACCOUNT 테이블

목적:
사용자 계좌 관리

기본 키:

* account_id

주요 정책:

* 사용자별 계좌명 중복 불가
* use_yn boolean 기준 비활성 처리
* current_balance 실시간 관리
* created_at, updated_at을 API 응답 기준으로 관리

---

# 9. CARD 테이블

목적:
카드 사용 관리

기본 키:

* card_id

주요 정책:

* 카드 사용 내역은 TRANSACTION에서 관리
* 카드 삭제 대신 use_yn=false 처리
* created_at, updated_at을 API 응답 기준으로 관리

---

# 10. CATEGORY 테이블

목적:
거래 분류 기준 제공

주요 정책:

* 단일 Depth 구조
* 하위 카테고리 미사용
* show=false 시 선택 목록 제외
* created_at, updated_at을 API 응답 기준으로 관리

---

# 11. ICON 테이블

목적:
계좌/카드/카테고리 공통 아이콘 관리

주요 정책:

* 사용자별 아이콘 관리
* show=false 시 선택 목록 제외
* created_at, updated_at을 API 응답 기준으로 관리

---

# 12. USER_SETTING 테이블

목적:
사용자별 앱 설정을 서버에 저장하고 재설치 또는 다른 기기에서 복원할 수 있도록 한다.

기본 키:

* setting_id

주요 컬럼:

| 컬럼 | 설명 |
|---|---|
| user_id | 사용자 ID |
| setting_key | 설정 키 |
| setting_value | JSON 설정 값 |
| created_at | 생성일시 |
| updated_at | 수정일시 |

주요 정책:

* `(user_id, setting_key)` unique
* 설정 저장 API는 같은 key가 있으면 갱신한다
* 민감정보, 토큰, 비밀번호는 저장하지 않는다

---

# 13. SYNC_CLIENT 테이블

목적:
오프라인 동기화를 요청하는 클라이언트 또는 기기를 식별한다.

기본 키:

* sync_client_id

주요 컬럼:

| 컬럼 | 설명 |
|---|---|
| user_id | 사용자 ID |
| client_id | 브라우저/앱 설치 단위 클라이언트 식별자 |
| device_name | 사용자가 식별 가능한 기기명 |
| platform | WEB/PWA/IOS/ANDROID 등 플랫폼 |
| last_synced_at | 마지막 동기화 성공 일시 |
| created_at | 생성일시 |
| updated_at | 수정일시 |

주요 정책:

* `(user_id, client_id)` unique
* 동기화 성공 시 last_synced_at 갱신
* 로그아웃해도 미동기화 Queue 확인 전 client_id를 삭제하지 않는다

---

# 14. SYNC_HISTORY 테이블

목적:
서버 반영 이력을 저장하여 중복 전송, 충돌, 실패 원인을 추적한다.

기본 키:

* sync_history_id

주요 컬럼:

| 컬럼 | 설명 |
|---|---|
| user_id | 사용자 ID |
| sync_client_id | 동기화 클라이언트 ID |
| transaction_id | 서버 거래 ID |
| client_temp_id | 클라이언트 임시 거래 ID |
| sync_action | CREATE/UPDATE/DELETE |
| sync_result | SUCCESS/FAILED/CONFLICT |
| error_message | 실패 또는 충돌 사유 |
| server_applied_at | 서버 반영 일시 |
| created_at | 이력 생성일시 |

주요 정책:

* 업로드 처리 결과마다 이력을 남긴다
* 중복 CREATE 요청은 기존 TRANSACTION을 반환하고 SUCCESS 이력을 추가할 수 있다
* 충돌 발생 시 CONFLICT 이력을 기록한다

---

# 15. REFRESH_TOKEN 테이블

목적:
서버 측 Refresh Token을 저장하여 로그아웃, 비밀번호 재설정, 회원탈퇴 시 전체 폐기를 지원한다. 다중 기기 로그인 상태를 관리한다.

기본 키:

* refresh_token_id

주요 컬럼:

| 컬럼 | 설명 |
|---|---|
| user_id | 사용자 ID |
| token_hash | 토큰 해시값 (원본 토큰은 저장하지 않는다) |
| user_agent | 요청 클라이언트 User-Agent |
| client_id | 클라이언트 식별자 (선택) |
| expires_at | 토큰 만료 일시 |
| revoked_at | 토큰 폐기 일시. NULL이면 유효 상태 |
| created_at | 생성일시 |
| updated_at | 수정일시 |

주요 정책:

* 로그인 성공 시 REFRESH_TOKEN 레코드를 생성하고 cookie로 발급한다
* 재발급 요청 시 token_hash를 검증하고 expires_at, revoked_at을 확인한다
* 로그아웃 시 해당 토큰의 revoked_at을 설정한다
* 비밀번호 재설정 성공 시 해당 사용자의 모든 REFRESH_TOKEN revoked_at을 설정한다
* 회원탈퇴 시 해당 사용자의 모든 REFRESH_TOKEN을 폐기한다
* 원본 토큰 값은 DB에 저장하지 않는다 (token_hash만 저장)

# 16. 인덱스 권장 사항

## TRANSACTION

권장 인덱스:

* user_id
* transaction_date
* category_id
* wallet_id
* wallet_type
* transaction_type
* client_temp_id
* sync_client_id
* synced_at

Unique 권장:

```text
user_id + sync_client_id + client_temp_id
```

# 17. 오프라인 동기화 고려사항

오프라인 상태에서 생성된 거래는 로컬 저장 후 재연결 시 서버와 동기화한다.

동기화 기준:

* created_at
* updated_at
* client_temp_id
* sync_client_id
* synced_at

중복 방지:

* 서버는 `(user_id, sync_client_id, client_temp_id)` 기준으로 동일 CREATE 요청을 중복 생성하지 않는다.
* 동일 요청이 이미 반영된 경우 기존 transaction_id를 반환한다.

충돌 발생 시:

* 최신 updated_at 우선 적용
* 충돌 또는 실패는 SYNC_HISTORY에 기록
