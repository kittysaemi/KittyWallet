# 유스케이스 정의

## 1. 문서 목적

본 문서는 미냥이 지갑 백엔드에서 구현할 주요 UseCase와 작성 기준을 정의한다.

UseCase는 하나의 사용자 행위 또는 API 기능을 처리하는 Application Layer의 핵심 단위이다.

---

# 2. UseCase 작성 기준

| 항목 | 기준 |
|---|---|
| 단위 | 하나의 명확한 기능 단위 |
| 위치 | 각 모듈 application/use-cases 하위 |
| 명명 | 동사 + 대상 + UseCase |
| 입력 | Command 또는 Query 객체 사용 |
| 출력 | Response DTO 또는 Result 객체 사용 |
| 책임 | 정책 검증, 처리 흐름 제어, Repository 호출 |

---

# 3. 공통 구조

```text
UseCase
 ├─ 입력값 수신
 ├─ 사용자 권한/소유자 검증
 ├─ 업무 정책 검증
 ├─ Repository 조회/저장
 ├─ 필요한 경우 DB Transaction 처리
 └─ 결과 반환
```

---

# 4. Auth UseCase

| UseCase | 입력 | 출력 | 설명 |
|---|---|---|---|
| SignUpUseCase | email, password | userId | 신규 회원 생성 |
| LoginUseCase | email, password | token 정보 | 로그인 및 토큰 발급 |
| RefreshTokenUseCase | refreshToken | accessToken | 토큰 재발급 |

| GetCurrentUserUseCase | userId | 사용자 정보 | 내 정보 조회 |

# 5. Transaction UseCase

## 5.1 CreateTransactionUseCase

| 항목 | 내용 |
|---|---|
| 목적 | 거래를 등록하고 계좌 잔액을 반영한다 |
| 입력 | user_id, transaction_type, wallet_type, wallet_id, category_id, amount, date, memo |

`wallet_id`는 `wallet_type=ACCOUNT`이면 `account_id`, `wallet_type=CARD`이면 `card_id`를 의미한다.
| 출력 | 생성된 거래 정보 |
| 주요 정책 | 금액 0 이하 불가, 미래 날짜 불가, 카테고리 필수, 비활성 계좌/카드 선택 불가, 카드 수입 거래 불가 |
| Transaction | 필수 |

처리 흐름:

```text
입력값 검증
 → 계좌/카드/카테고리 존재 여부 확인
 → transaction_type=INCOME, wallet_type=CARD 조합 차단
 → 사용자 소유 데이터 확인
 → 거래 생성
 → wallet_type=ACCOUNT이면 계좌 잔액 반영
 → 결과 반환
```

## 5.2 UpdateTransactionUseCase

| 항목 | 내용 |
|---|---|
| 목적 | 거래를 수정하고 잔액을 재계산한다 |
| 주요 정책 | 기존 반영 금액 복구 후 변경 금액 기준 재계산 |
| Transaction | 필수 |

처리 흐름:

```text
원본 거래 조회
 → 삭제 여부 확인
 → 기존 잔액 반영분 복구
 → 변경값 검증
 → 거래 수정
 → 변경 잔액 반영
```

## 5.3 DeleteTransactionUseCase

| 항목 | 내용 |
|---|---|
| 목적 | 거래를 논리 삭제하고 잔액을 복구한다 |
| 주요 정책 | deleted_yn=true 처리, 계좌 잔액 복구 |
| Transaction | 필수 |

## 5.4 SearchTransactionsUseCase

| 항목 | 내용 |
|---|---|
| 목적 | 기간, 키워드, 계좌, 카드, 카테고리 기준으로 거래를 검색한다 |
| 제외 기준 | deleted_yn=true 제외 |
| 정렬 | 거래일 최신순 |

## 5.5 GetTransactionDetailUseCase

| 항목 | 내용 |
|---|---|
| 목적 | 단일 거래 상세 정보를 조회한다 |
| 입력 | user_id, transaction_id |
| 출력 | 거래 상세 정보 |
| 주요 정책 | 사용자 소유 확인, deleted_yn=true 거래 제외 |

## 5.6 GetRecentTransactionsUseCase

| 항목 | 내용 |
|---|---|
| 목적 | 대시보드 또는 홈 화면용 최근 거래를 조회한다 |
| 입력 | user_id, limit |
| 출력 | 최근 거래 목록 |
| 주요 정책 | deleted_yn=true 제외, 거래일 최신순, limit 1~20 |

# 11. Sync UseCase

| UseCase | 설명 | 기준 |
|---|---|---|
| SyncTransactionsUseCase | 오프라인 거래 동기화 | `(user_id, sync_client_id, client_temp_id)` 중복 방지 |
| RegisterSyncClientUseCase | 동기화 클라이언트 등록 또는 갱신 | `(user_id, client_id)` 기준 upsert |
| RecordSyncHistoryUseCase | 동기화 이력 기록 | SUCCESS/FAILED/CONFLICT 모두 기록 |
| RetryFailedSyncUseCase | 실패 데이터 재시도 | 실패 항목만 재처리 |

---

# 12. Settings UseCase

| UseCase | 설명 | 기준 |
|---|---|---|
| GetSettingsUseCase | 사용자 설정 조회 | USER_SETTING에서 사용자별 조회 |
| UpdateSettingsUseCase | 사용자 설정 저장 | `(user_id, setting_key)` 기준 upsert |

설정 UseCase는 Access Token, Refresh Token, Password 등 민감정보를 저장하지 않는다.

---

# 13. UseCase 금지 사항

* Request 객체 직접 사용 금지
* Prisma 직접 호출 금지
* Controller 전용 DTO에 강결합 금지
* 여러 기능을 하나의 UseCase에 과도하게 통합 금지
* 정책 문서와 다른 임의 규칙 추가 금지
