# 모듈 정의

## 1. 문서 목적

본 문서는 미냥이 지갑 백엔드에서 사용하는 기능별 모듈과 각 모듈의 책임 범위를 정의한다.

모듈은 기능 변경 시 영향 범위를 최소화하기 위해 명확한 책임 단위로 분리한다.

---

# 2. 모듈 분리 기준

| 기준 | 설명 |
|---|---|
| 기능 단위 | 인증, 거래, 계좌 등 독립 기능 기준 분리 |
| 데이터 책임 | 주요 테이블 또는 도메인 기준 분리 |
| 정책 책임 | 정책이 다른 기능은 별도 모듈로 분리 |
| 변경 영향 | 변경 빈도가 다른 기능은 분리 |
| 재사용성 | 공통 기능은 common 또는 infra로 분리 |

---

# 3. 전체 모듈 목록

| 모듈 | 책임 |
|---|---|
| AuthModule | 인증, 회원가입, 로그인, 토큰 처리 |
| UserModule | 사용자 정보 조회 및 사용자 상태 관리 |
| SettingsModule | 사용자 설정 조회 및 저장 |
| TransactionModule | 거래 등록, 수정, 삭제, 검색 |
| AccountModule | 계좌 등록, 수정, 사용 여부 관리 |
| CardModule | 카드 등록, 수정, 사용 여부 관리 |
| CategoryModule | 카테고리 등록, 수정, 표시 여부 관리 |
| IconModule | 아이콘 등록, 조회, 사용 여부 관리 |
| StatisticsModule | 월별·카테고리별 통계 조회 |
| SyncModule | 오프라인 저장 데이터 동기화 |
| HealthModule | 서버 상태 확인 |

---

# 4. AuthModule

## 4.1 책임

* 회원가입
* 로그인
* Access Token 발급
* Refresh Token 발급 및 갱신
* 로그아웃
* 인증 사용자 식별

## 4.2 주요 UseCase

| UseCase | 설명 |
|---|---|
| SignUpUseCase | 신규 사용자 생성 |
| LoginUseCase | 이메일/비밀번호 인증 |
| RefreshTokenUseCase | Access Token 재발급 |
| LogoutUseCase | 토큰 무효화 또는 클라이언트 상태 초기화 기준 제공 |

## 4.3 참조 정책

* 로그인정책
* 보안정책
* 백엔드예외처리

---

# 5. UserModule

## 5.1 책임

* 내 사용자 정보 조회
* 사용자 기본 정보 관리
* 사용자 기준 데이터 분리 기준 제공

## 5.2 주요 API

| 기능 | API |
|---|---|
| 내 정보 조회 | GET /api/v1/users/me |

---

# 6. SettingsModule

## 6.1 책임

* 사용자 설정 조회
* 사용자 설정 저장
* 설정 key/value 검증
* USER_SETTING upsert 처리

## 6.2 주요 API

| 기능 | API |
|---|---|
| 설정 조회 | GET /api/v1/settings |
| 설정 저장 | PUT /api/v1/settings |

---

# 7. TransactionModule

## 7.1 책임

* 수입 거래 등록
* 지출 거래 등록
* 카드 결제수단 거래 등록
* 거래 수정
* 거래 삭제
* 거래 상세 조회
* 거래 검색
* 계좌 거래 기준 잔액 반영

## 7.2 주요 UseCase

| UseCase | 설명 |
|---|---|
| CreateTransactionUseCase | 거래 등록 및 잔액 반영 |
| UpdateTransactionUseCase | 기존 금액 복구 후 재계산 |
| DeleteTransactionUseCase | 논리 삭제 및 잔액 복구 |
| SearchTransactionsUseCase | 기간, 키워드, 필터 검색 |
| GetTransactionDetailUseCase | 거래 상세 조회 |

## 7.3 참조 정책

* 거래정책
* 트랜잭션정책
* 예외처리

# 12. StatisticsModule

## 12.1 책임

* 월별 수입/지출 통계
* 카테고리별 소비 통계
* 최근 거래 기반 요약
* deleted_yn=true 데이터 제외

## 12.2 처리 기준

통계는 거래 데이터를 기준으로 산출하며, 거래 저장·수정·삭제 후 조회 시 최신 데이터가 반영되어야 한다.

---

# 13. SyncModule

## 13.1 책임

* 오프라인 임시 저장 거래 수신
* 클라이언트 임시 ID 기준 중복 방지
* updated_at 기준 충돌 처리
* 부분 실패 데이터 재시도 기준 제공
* SYNC_CLIENT 등록 및 last_synced_at 갱신
* SYNC_HISTORY 기록

---

# 14. 모듈 간 참조 원칙

* 다른 모듈의 Repository를 직접 참조하지 않는다.
* 필요한 경우 공개 Service 또는 UseCase 인터페이스를 통해 접근한다.
* 공통 유효성 검증은 별도 Domain Service 또는 Policy로 분리한다.
* 순환 참조가 발생하면 모듈 경계를 재검토한다.
