# 인증 인가 구조

## 1. 문서 목적

본 문서는 미냥이 지갑 백엔드의 인증 및 인가 처리 구조를 정의한다.

인증은 JWT 기반으로 처리하며, 사용자 데이터는 user_id 기준으로 분리한다.

---

# 2. 인증 방식

| 항목 | 기준 |
|---|---|
| 로그인 방식 | 이메일 + 비밀번호 |
| 인증 방식 | JWT |
| Access Token | API 인증에 사용, 만료 시간 2시간 |
| Refresh Token | Access Token 재발급에 사용, 만료 시간 30일 |
| 비밀번호 저장 | bcrypt 해시 저장 |
| 사용자 식별 | user_id 기준 |
| Refresh Token Rotation | 갱신 시마다 새 Refresh Token 발급, 기존 토큰 폐기 |

---

# 3. 인증 흐름

## 3.1 로그인

```text
로그인 요청
 → 이메일 사용자 조회
 → 비밀번호 검증
 → Access Token 발급 (response body)
 → Refresh Token 발급 → DB 저장 → Set-Cookie (httpOnly Secure SameSite=Strict)
```

## 3.2 인증 API 요청

```text
API 요청
 → Authorization Header 확인
 → Access Token 검증
 → user_id 추출
 → CurrentUser로 Controller 전달
 → UseCase 실행
```

## 3.3 토큰 갱신 (Rotation 적용)

```text
Access Token 만료
 → /auth/refresh 요청 (Refresh Token은 cookie로 자동 전송)
 → Refresh Token 검증 (DB 조회, revoked_at 확인)
 → 기존 Refresh Token 폐기 (revoked_at 설정)
 → 새 Access Token 발급 (response body)
 → 새 Refresh Token 발급 → DB 저장 → Set-Cookie (httpOnly Secure SameSite=Strict)
 → 기존 요청 재시도
```

## 3.4 로그아웃

```text
/auth/logout 요청 (Refresh Token cookie 자동 전송)
 → DB에서 Refresh Token 폐기
 → Set-Cookie로 cookie 만료 처리 (Max-Age=0)
```

---

# 4. Guard 구성

| Guard | 설명 |
|---|---|
| JwtAuthGuard | Access Token 검증 |
| OptionalAuthGuard | 선택 인증이 필요한 경우 사용 |
| OwnershipGuard | 필요 시 리소스 소유자 확인 |

기본적으로 로그인 이후 API는 JwtAuthGuard를 적용한다.

---

# 5. Public API

인증 없이 접근 가능한 API는 명시적으로 관리한다.

| API | 설명 |
|---|---|
| POST /api/v1/auth/signup | 회원가입 |
| POST /api/v1/auth/login | 로그인 |
| POST /api/v1/auth/refresh | 토큰 재발급 |
| GET /health | 서버 상태 확인 |

---

# 6. 인가 기준

모든 사용자 데이터는 user_id 기준으로 분리한다.

적용 대상:

* 거래
* 계좌
* 카드
* 카테고리
* 아이콘
* 통계
* 동기화 데이터

Repository 조회 조건에는 user_id를 포함해야 한다.

---

# 7. 비밀번호 처리

| 항목 | 기준 |
|---|---|
| 저장 | bcrypt 해시 저장 |
| 비교 | bcrypt compare 사용 |
| 평문 저장 | 금지 |
| 로그 출력 | 금지 |

---

# 8. 토큰 Payload 기준

Access Token Payload:

```json
{
  "sub": "user_id",
  "email": "user@example.com"
}
```

Refresh Token은 MVP 기준으로 DB에 저장(user_id 매핑)하며, 서버가 유효성 검증 및 폐기 처리를 담당한다.

Refresh Token 서버 저장 방식:

| 항목 | 기준 |
|---|---|
| 저장 위치 | DB (`REFRESH_TOKEN` 테이블 또는 `USER` 테이블의 컬럼) |
| 저장 값 | 해시 처리된 토큰 값 |
| 폐기 시점 | 로그아웃, 비밀번호 재설정 시 전체 폐기 |
| 클라이언트 전달 | `httpOnly Secure SameSite=Strict` cookie로 발급 |

---

# 9. 인증 예외 처리

| 조건 | 처리 |
|---|---|
| Access Token 없음 | 401 반환 |
| Access Token 만료 | 401 반환, 프론트에서 refresh 요청 |
| Refresh Token 만료 | 로그인 화면 이동 |
| 비밀번호 불일치 | 로그인 실패 메시지 |
| 사용자 없음 | 로그인 실패 메시지 |

---

# 10. 보안 기준

* 비밀번호 평문 저장 금지
* 민감정보 로그 출력 금지
* 인증 없는 사용자 데이터 접근 차단
* CORS 허용 도메인 제한
* 운영 환경에서 JWT Secret 분리
* Refresh Token 탈취 대응을 고려한 저장 방식 사용
