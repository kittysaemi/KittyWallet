# 백엔드 아키텍처

## 1. 문서 목적

본 문서는 미냥이 지갑 백엔드의 전체 아키텍처 구조와 계층 간 책임, 의존성 방향, 개발 기준을 정의한다.

백엔드는 단순 API 구현이 아니라 정책 기반 기능 처리, 데이터 무결성 보장, API 변경 대응, 장기 유지보수를 목표로 설계한다.

---

# 2. 아키텍처 목표

|---|---|
| 구조 기준 | 기능 중심 모듈 구조 |

---

# 3. 전체 아키텍처 구성

```text
Client
  ↓
Controller
  ↓
UseCase / Service
  ↓
Repository Interface
  ↓
Repository Implementation
  ↓
Prisma
  ↓
PostgreSQL
```

각 계층은 위에서 아래 방향으로만 의존한다.

하위 계층은 상위 계층을 직접 참조하지 않는다.

---

# 4. 계층별 책임

| 계층 | 책임 |
|---|---|
| Controller | HTTP 요청 수신, DTO 검증, 인증 사용자 전달 |
| UseCase | 비즈니스 정책 처리, 트랜잭션 흐름 제어 |
| Domain | 핵심 규칙, 상태 판단, 계산 로직 |
| Repository | 데이터 저장소 접근 추상화 |
| Infrastructure | Prisma, 외부 모듈, 암호화, 토큰 처리 |
| Common | 공통 응답, 예외, 유틸, 데코레이터 |

---

# 5. 의존성 방향

```text
Controller → UseCase → Repository Interface
Infrastructure → Repository Interface 구현
Common → 전 계층에서 참조 가능
```

## 금지 기준

* Controller에서 Prisma 직접 호출 금지
* Controller에서 잔액 계산 등 비즈니스 로직 처리 금지
* UseCase에서 HTTP Request 객체 직접 의존 금지
* Repository에서 응답 메시지 생성 금지
* Prisma Entity를 API Response로 직접 반환 금지

---

# 6. 모듈 구성 방향

백엔드는 기능 단위 모듈을 기준으로 구성한다.

| 모듈 | 설명 |
|---|---|
| auth | 회원가입, 로그인, 토큰 갱신, 로그아웃 |
| user | 사용자 정보 조회 및 관리 |
| transaction | 거래 등록, 수정, 삭제, 검색 |
| account | 계좌 등록, 수정, 사용 여부 관리 |
| card | 카드 등록, 수정, 사용 여부 관리 |
| category | 카테고리 등록, 수정, 표시 여부 관리 |
| icon | 아이콘 등록, 조회, 사용 여부 관리 |
| statistics | 통계 조회 및 집계 |
| sync | 오프라인 데이터 동기화 처리 |

# 7. 요청 처리 흐름

## 7.1 일반 조회 흐름

```text
Request
 → Auth Guard
 → Controller
 → Query DTO 검증
 → UseCase 실행
 → Repository 조회
 → Response DTO 변환
 → 공통 응답 반환
```

## 7.2 저장 처리 흐름

```text
Request
 → Auth Guard
 → Controller
 → Body DTO 검증
 → UseCase 실행
 → 정책 검증
 → DB Transaction 처리
 → Response DTO 변환
 → 공통 응답 반환
```

---

# 8. 주요 설계 원칙

## 8.1 정책 우선

거래 저장, 계좌 잔액 계산, 삭제 처리, 오프라인 동기화는 기존 정책 문서를 기준으로 구현한다.

## 8.2 상태 기반 처리

API 실패, 인증 만료, 데이터 없음, 오프라인 동기화 실패는 공통 상태 및 예외 처리 기준에 맞춘다.

## 8.3 데이터 무결성 우선

거래 저장·수정·삭제 시 계좌 잔액과 거래 데이터가 불일치하지 않도록 DB Transaction을 사용한다.

## 8.4 API 변경 영향 최소화

API 응답은 Response DTO와 공통 응답 규격으로 감싸며, Prisma 모델을 외부로 직접 노출하지 않는다.

# 10. 최종 기준

백엔드 아키텍처는 다음 기준을 만족해야 한다.

* 기능별 모듈 분리
* Controller와 비즈니스 로직 분리
* UseCase 중심 정책 처리
* Repository 기반 DB 접근
* Prisma 직접 노출 금지
* 공통 응답 및 예외 처리 적용
* 트랜잭션 기반 데이터 무결성 보장
* 향후 API 및 기능 변경에 대응 가능한 구조 유지
