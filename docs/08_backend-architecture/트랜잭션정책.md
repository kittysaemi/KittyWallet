# 트랜잭션 정책

## 1. 문서 목적

본 문서는 미냥이 지갑 백엔드에서 DB Transaction을 적용해야 하는 기준과 거래 데이터 처리 흐름을 정의한다.

거래 등록, 수정, 삭제는 계좌 잔액 및 카드 사용 내역과 연결되므로 데이터 무결성을 최우선으로 처리한다.

---

# 2. 기본 원칙

| 항목 | 정책 |
|---|---|
| 적용 기준 | 두 개 이상 데이터 변경 시 Transaction 사용 |
| 실패 처리 | 하나라도 실패하면 전체 Rollback |
| 잔액 처리 | 계좌 거래는 거래 데이터와 계좌 잔액을 함께 처리 |
| 카드 처리 | 카드 거래는 카드 사용 내역으로 저장하고 계좌 잔액은 변경하지 않음 |
| 동기화 | 부분 실패 항목은 개별 실패로 관리 |

---

# 3. Transaction 필수 대상

| 기능 | 사유 |
|---|---|
| 계좌 거래 등록 | 거래 생성 + 계좌 잔액 반영 |
| 계좌 거래 수정 | 기존 잔액 복구 + 거래 수정 + 새 잔액 반영 |
| 계좌 거래 삭제 | deleted_yn 처리 + 계좌 잔액 복구 |
| 카드 거래 등록/수정/삭제 | 거래 내역 변경. 계좌 잔액 변경 없음 |
| 오프라인 동기화 | 여러 거래 일괄 반영 |
| 계정 삭제 또는 비활성 일괄 처리 | 연관 데이터 상태 변경 |

---

# 4. 거래 등록 Transaction

```text
거래 등록 요청
 → 입력값 검증
 → 계좌/카드/카테고리 검증
 → DB Transaction 시작
   → 거래 생성
   → wallet_type=ACCOUNT이면 계좌 잔액 반영
   → wallet_type=CARD이면 계좌 잔액 변경 없음
 → Commit
 → 응답 반환
```

실패 시:

```text
거래 생성 실패 또는 잔액 반영 실패
 → Rollback
 → 저장 실패 응답
```

---

# 5. 거래 수정 Transaction

```text
거래 수정 요청
 → 원본 거래 조회
 → 삭제 여부 확인
 → DB Transaction 시작
   → 기존 거래 금액 기준 잔액 복구
   → 거래 정보 수정
   → 수정 거래 기준 잔액 재반영
 → Commit
```

주의:

* 원본 거래 조회 실패 시 수정 불가
* deleted_yn=true 거래는 수정 불가
* wallet_type 또는 wallet_id 변경 시 계좌 거래에 해당하는 기존 참조 대상과 새 참조 대상의 잔액 조정 필요
* wallet_id는 wallet_type에 따라 account_id 또는 card_id를 참조한다.
* transaction_type=INCOME, wallet_type=CARD 조합은 수정/등록 모두 불가하다.

---

# 6. 거래 삭제 Transaction

```text
거래 삭제 요청
   → 원본 거래 조회
   → DB Transaction 시작
   → wallet_type=ACCOUNT이면 계좌 잔액 복구
   → wallet_type=CARD이면 계좌 잔액 변경 없음
   → 거래 deleted_yn=true 처리
 → Commit
```

---

# 7. 잔액 계산 기준

| 조합 | 처리 |
|---|---|
| transaction_type=INCOME, wallet_type=ACCOUNT | 계좌 잔액 증가 |
| transaction_type=EXPENSE, wallet_type=ACCOUNT | 계좌 잔액 감소 |
| transaction_type=EXPENSE, wallet_type=CARD | 카드 거래로 저장, 계좌 잔액 변경 없음 |
| transaction_type=INCOME, wallet_type=CARD | 저장 불가 |
| 삭제 | 기존 반영 금액 반대로 복구 |

---

# 8. Prisma Transaction 기준

Prisma `$transaction`을 사용한다.

```text
prisma.$transaction(async (tx) => {
  // 거래 저장
  // 잔액 변경
})
```

Transaction 내부에서는 동일한 tx 객체를 사용한다.

---

# 9. 금지 사항

* 거래 저장 후 별도 요청으로 잔액 변경 금지
* 잔액 변경 실패 상태로 거래만 저장 금지
* Transaction 내부에서 외부 API 호출 금지
* 긴 작업을 Transaction 내부에서 처리 금지
* 실패 후 수동 보정에 의존하는 구조 금지
