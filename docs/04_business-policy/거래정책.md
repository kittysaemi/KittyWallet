# 거래정책

## 1. 문서 목적

본 문서는 거래(내역) 등록, 수정, 삭제, 조회 처리 기준을 정의한다.

---

# 2. 거래 기본 정책

| 항목    | 정책                 |
| ----- | ------------------ |
| 거래 유형 | 수입 / 지출 구분 필수 |
| 지갑 유형 | 계좌 / 카드 결제수단 구분 필수 |
| 금액 입력 | 숫자만 입력 가능          |
| 금액 조건 | 0 이하 입력 불가         |
| 잔액 음수  | 계좌별 마이너스 허용/한도 정책에 따라 날짜별 잔액 흐름이 허용 범위 아래로 내려가면 저장 불가 |
| 메모 입력 | 선택 입력              |
| 날짜 입력 | 직접 선택 가능           |
| 미래 날짜 | 등록 불가 (사용자 설정 timezone 기준 today 이후 날짜 차단. 기본값 `Asia/Seoul`) |

---

# 3. 거래 유형 및 지갑 선택 정책

| 항목 | 정책 |
|---|---|
| transaction_type | `INCOME` 또는 `EXPENSE`만 사용 |
| wallet_type | `ACCOUNT` 또는 `CARD`만 사용 |
| 결제수단 구조 | 계좌와 카드는 거래 유형이 아니라 결제수단이다 |
| 카드 수입 | 카드는 수입을 지원하지 않으므로 `transaction_type=INCOME`, `wallet_type=CARD` 조합은 저장 불가 |
| 카드 지출 | 카드 거래는 `transaction_type=EXPENSE`, `wallet_type=CARD`로 저장 |
| 계좌 거래 | 계좌 거래는 `transaction_type=INCOME|EXPENSE`, `wallet_type=ACCOUNT`로 저장 |
| 화면 입력 | 거래 등록은 한 화면에서 처리하되, 수입 선택 시 카드 선택을 비활성화한다 |
| 내역 관리 | 계좌 거래와 카드 거래 내역은 `wallet_type` 기준으로 분리 조회 및 관리할 수 있다 |

---

# 4. 저장 정책

| 항목    | 정책                  |
| ----- | ------------------- |
| 계좌 거래 저장 | 계좌 잔액 자동 반영         |
| 카드 거래 저장 | 카드 거래 내역으로 저장하고 계좌 잔액은 변경하지 않음 |
| 거래 수정 | 기존 반영 금액 복구 후 변경 기준 재계산. 날짜별 잔액 흐름 검증 후 저장 |
| 거래 삭제 | 계좌 거래는 잔액 복구, 카드 거래는 내역만 삭제 처리. 수입 삭제 시에만 날짜별 잔액 흐름 검증 (지출 삭제는 잔액 복구 방향이므로 검증 생략) |
| 삭제 처리 | deleted_yn 기반 논리 삭제 |

## 4.1 계좌 잔액 검증 정책

계좌 거래 등록/수정 및 수입 거래 삭제는 저장 전 다음 기준으로 날짜별 잔액 흐름을 검증한다.

1. 계좌의 `initial_balance`에서 시작한다.
2. 삭제되지 않은 계좌 거래를 `transaction_date` 기준으로 일자별 합산한다.
3. 같은 날짜의 여러 거래는 일별 합산액으로 한 번 검증한다.
4. 마이너스 미허용 계좌는 어느 날짜의 잔액도 0 미만이 될 수 없다.
5. 마이너스 허용 계좌는 어느 날짜의 잔액도 `-negative_balance_limit`보다 작아질 수 없다.

거래 날짜를 과거로 변경하거나 금액, 거래 유형, 계좌를 변경해 중간 날짜의 잔액이 허용 범위 아래로 내려가면 수정은 차단한다.

**지출 거래 삭제**는 잔액을 증가시키는 방향이므로 날짜별 잔액 흐름 검증을 생략한다.

**수입 거래 삭제**는 잔액을 감소시켜 기존 지출이 마이너스로 노출될 수 있으므로 삭제 전 날짜별 잔액 흐름을 검증한다. 삭제 후 어느 날짜의 잔액이 허용 범위 아래로 내려가면 삭제를 차단한다. 이 경우 관련 지출을 먼저 삭제하거나 수정한 뒤 수입을 삭제해야 한다.

# 7. 데이터 무결성 정책

| 항목                  | 정책    |
| ------------------- | ----- |
| 존재하지 않는 wallet_id   | 저장 불가 |
| 존재하지 않는 category_id | 저장 불가 |
| 비활성 계좌/카드           | 선택 불가 |
| deleted_yn=true     | 수정 불가 |
| 카드 수입 거래             | 저장 불가 |
| 날짜별 잔액 한도 초과        | 저장 불가 |

## 7.1 아카이브된 지갑의 거래 처리

계좌 또는 카드가 아카이브(`deleted_yn=true`)된 경우, 해당 지갑에 연결된 거래는 다음 정책을 따른다.

| 항목 | 정책 |
|---|---|
| 조회 | 가능. 거래 목록 및 상세에서 `wallet_deleted=true`로 표시 |
| 수정 | 불가. 수정 시도 시 `WALLET_001` 에러 반환 |
| 삭제 | 가능 |
| 신규 등록 | 불가. 아카이브된 지갑은 거래 등록 지갑 선택 목록에서 제외 |

# 8. 카드할부 정책

## 8.1 할부 적용 기준

| 항목 | 정책 |
|---|---|
| 할부 대상 | wallet_type=CARD인 지출 거래에서만 사용 |
| 할부 데이터 구조 | 부모 할부(CARD_INSTALLMENT) + 월별 거래(TRANSACTION) |
| 통계 반영 기준 | 월별 TRANSACTION.amount 기준. 부모 할부는 통계/지출 집계에 반영하지 않음 |
| 할부 거래 구분 | TRANSACTION.installment_id 존재 여부로 구분 |
| 계좌 잔액 | 카드 거래이므로 변경 없음 |

## 8.2 금액 계산 정책

| 항목 | 정책 |
|---|---|
| original_amount | 최초 등록한 구매금액. 회차별 거래 금액 수정 후에도 변경하지 않음 |
| 현재 총 청구금액 | DB에 저장하지 않음. 상세 조회 시 회차별 TRANSACTION.amount 합계로 계산 |
| 회차별 금액 분배 | 구매금액 ÷ 할부 개월 수. 나머지는 1회차에 합산 |
| 회차 수정 | 저장 후 생성된 월별 거래에서 각 회차 금액 개별 수정 가능 |
| 통계 반영 | 수정된 월별 거래 금액 기준으로 계산 |

## 8.3 생성 정책

| 항목 | 정책 |
|---|---|
| 첫 회차 날짜 | 구매일(purchase_date) 기준 |
| 이후 회차 날짜 | 구매일 기준 월 단위 증가 |
| 생성 단위 | 부모 할부 1건 + 월별 거래 N건(할부 개월 수) |

## 8.4 수정 제약

| 항목 | 정책 |
|---|---|
| 지갑(결제수단) 변경 | 이미 저장된 할부 회차 거래는 지갑 변경 불가. 수정 시도 시 안내 메시지 표시 |
| 안내 메시지 | "할부 거래는 결제 수단을 변경할 수 없습니다. 전체 삭제 후 다시 등록해 주세요." |
| 대안 | 부모 할부 전체 삭제(연결 회차 거래 포함 소프트 삭제) 후 올바른 카드로 재등록 |
| 등록 화면에서의 지갑 변경 | 저장 전 카드 → 계좌로 변경 시에는 할부 설정만 초기화. DB 영향 없음 |

## 9. 영수증 OCR 거래 초안 정책

| 항목 | 정책 |
|---|---|
| 적용 범위 | 신규 거래 등록 시작 시에만 사용, 거래 수정에는 미제공 |
| 등록 방식 | 하단 `+`에서 수동 입력/카메라 입력/사진 입력/텍스트 입력. 카메라 입력은 모바일 기기에서만 제공하며 PC 웹에서는 제외 |
| 저장 시점 | OCR 결과만으로 거래를 생성하지 않으며 사용자가 거래폼을 확인해 저장 |
| 초안 항목 | 날짜, 총액, 상호명, 상품명 후보 |
| 메모 | 상품명 후보를 쉼표로 연결한 메모 초안만 반영 |
| 제외 항목 | 계좌, 카드, 카테고리 자동 추천·변경 금지 |
| 모호한 결과 | 자동 확정하지 않고 경고를 표시해 사용자가 수정 |

현재 비상용 개발 단계에서는 OCR 원문 또는 텍스트 입력 원문과 사용자가 최종 저장한 거래값을 문자 파서 학습 표본으로 사용할 수 있다. 영수증 원본 이미지는 저장하지 않는다. 학습 표본은 즉시 모델을 변경하는 입력값이 아니라, 검증 후 재학습에 사용하는 데이터셋이다. 상용화 시에는 별도 동의와 보관 정책을 정하기 전까지 이 수집을 활성화하지 않는다.
