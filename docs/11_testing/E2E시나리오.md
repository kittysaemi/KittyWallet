# E2E 시나리오

## 1. 문서 목적

본 문서는 미냥이 지갑의 사용자 전체 흐름을 기준으로 End-to-End 테스트 시나리오를 정의한다.

E2E 시나리오는 실제 사용자가 모바일 웹앱에서 수행하는 행동을 기준으로 작성하며, Playwright, Cypress 등 자동화 도구로 변환 가능한 수준의 절차와 기대 결과를 포함한다.

---

## 2. E2E 테스트 기준

| 항목 | 기준 |
|---|---|
| 실행 환경 | 모바일 뷰포트 우선 |
| 기본 경로 | `/login`에서 시작 |
| 인증 | 테스트 계정 사용 |
| 데이터 | 테스트 전용 DB 또는 fixture 사용 |
| 네트워크 | 온라인/오프라인 모두 검증 |
| 검증 방식 | URL, 화면 텍스트, API 결과, 상태 표시 확인 |

---

## 3. 권장 모바일 뷰포트

| 기기 | 크기 |
|---|---|
| iPhone SE | 375 x 667 |
| iPhone 14 | 390 x 844 |
| Android 기준 | 412 x 915 |

---

## 4. 테스트 데이터 초기화 원칙

- 각 E2E 시나리오는 독립 실행 가능해야 한다.
- 테스트 시작 전 사용자, 계좌, 카드, 카테고리, 거래 데이터를 초기화한다.
- 테스트 종료 후 생성 데이터는 삭제하거나 테스트 DB를 리셋한다.
- 오프라인 테스트는 Sync Queue와 IndexedDB 초기화가 필요하다.

# 5. 핵심 사용자 흐름

## E2E-CORE-001 회원가입 → 로그인 → 대시보드 진입

| 항목 | 내용 |
|---|---|
| 목적 | 신규 사용자가 가입 후 로그인하여 대시보드에 진입하는 흐름 검증 |
| 우선순위 | Critical |

### 절차

1. `/signup` 진입
2. 이메일 입력
3. 비밀번호 입력
4. 비밀번호 확인 입력
5. 회원가입 버튼 클릭
6. `/login` 이동 확인
7. 이메일/비밀번호 입력
8. 로그인 버튼 클릭
9. `/dashboard` 이동 확인

### 기대 결과

- 회원가입이 성공한다.
- 로그인 후 대시보드로 이동한다.
- 사용자 카드가 표시된다.
- 최근 거래 목록 영역이 표시된다.

### 자동화 확인 항목

- URL: `/dashboard`
- 사용자 닉네임 표시
- 인증 토큰 저장 여부
- 대시보드 API 호출 성공 여부

## E2E-CORE-002 로그인 → 계좌 등록 → 거래 등록 → 대시보드 반영

| 항목 | 내용 |
|---|---|
| 목적 | 계좌 기반 지출 거래 등록 후 대시보드와 잔액이 반영되는지 검증 |
| 우선순위 | Critical |

### 절차

1. `/login` 진입
2. 테스트 계정 로그인
3. `/accounts` 이동
4. 계좌 등록 버튼 클릭
5. 계좌명 입력
6. 초기 잔액 입력
7. 아이콘 선택
8. 저장 클릭
9. `/transactions/new` 이동
10. 거래 유형 `지출` 선택
11. 금액 입력
12. 오늘 날짜 선택
13. 방금 등록한 계좌 선택
14. 카테고리 선택
15. 저장 클릭
16. `/dashboard` 이동

### 기대 결과

- 계좌가 목록에 표시된다.
- 거래가 등록된다.
- 최근 거래 목록에 해당 거래가 표시된다.
- 계좌 잔액이 지출 금액만큼 차감된다.

### 자동화 확인 항목

- 계좌 목록에 계좌명 존재
- 최근 거래 목록에 메모/금액 표시
- 대시보드 자산 요약 변경

7. 등록한 카드 선택
8. 카테고리 선택
9. 금액 입력

### 기대 결과

- 통계 화면에 지출 금액이 반영된다.
- 계좌 잔액은 변경되지 않는다.

- 거래 금액이 변경된다.
- 계좌 잔액이 새 금액 기준으로 재계산된다.
- 통계가 새 금액 기준으로 반영된다.

1. 로그인
2. 계좌 등록 화면 이동

4. 해당 계좌로 지출 거래 300000 등록 (초기잔액 0 → 잔액 -300000, 한도 도달)
5. 추가 지출 1 등록 시도 (한도 초과)

- 한도 초과 지출 등록 및 한도 초과 수정은 ACCOUNT_004로 거부된다.
- 오류 메시지가 화면에 표시된다.

# 7. 오프라인/PWA 흐름

## E2E-OFFLINE-001 오프라인 거래 등록 → 온라인 복구 동기화

| 항목 | 내용 |
|---|---|
| 목적 | 오프라인 상태에서 등록한 거래가 온라인 복구 후 서버에 동기화되는지 검증 |
| 우선순위 | Critical |

### 절차

1. 로그인
2. 계좌/카테고리 데이터 조회하여 캐시 생성
3. 네트워크 오프라인 전환
4. `/transactions/new` 이동
5. 거래 입력
6. 저장 클릭
7. 오프라인 저장 상태 확인
8. 네트워크 온라인 전환
9. 동기화 완료 확인
10. 거래 목록 새로고침

### 기대 결과

- 오프라인 상태에서 거래가 로컬 저장된다.
- Sync Queue에 pending 상태로 등록된다.
- 온라인 복구 후 syncing 상태가 표시된다.
- 서버 동기화 성공 후 success 상태가 된다.
- 거래 목록에 서버 반영된 거래가 표시된다.

4. 동일 화면 재진입

### 기대 결과

- 캐시된 최근 데이터가 표시된다.
- offline 상태 안내가 표시된다.
- 온라인 전용 기능은 제한된다.

---

3. 온라인 전환

# 10. E2E 자동화 코드 생성 규칙

AI가 본 문서를 기반으로 테스트 코드를 생성할 때 다음 기준을 따른다.

- 시나리오 ID를 테스트명에 포함한다.
- fixture 데이터는 시나리오별로 독립 생성한다.
- API mock 여부를 명시한다.
- 모바일 뷰포트를 기본값으로 설정한다.
- 실패 시 screenshot, trace, console log를 저장한다.
- 오프라인 테스트는 브라우저 context의 network 상태를 제어한다.
- IndexedDB와 localStorage는 테스트 전 초기화한다.
