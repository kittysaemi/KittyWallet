## OP-CHECK-005

### 항목

DB Migration 검증

### 수행 방법

운영 배포 전 staging PostgreSQL 데이터베이스가 있으면 현재 main 기준 migration을 적용한다.

staging DB를 별도로 운영하지 않는 경우에는 다음 대체 검증을 모두 수행한다.

1. 비어 있는 PostgreSQL 데이터베이스에서 전체 migration과 seed를 적용한다.
2. 운영 DB에는 파괴적 SQL을 직접 실행하지 않고 read-only 조회로 현재 schema 상태만 확인한다.
3. 운영 health와 DB health를 확인한다.

```bash
npx prisma migrate deploy --schema apps/backend/prisma/schema.prisma
npx prisma migrate status --schema apps/backend/prisma/schema.prisma
npm run db:seed
```

회원탈퇴 물리 삭제와 관련된 컬럼 제거 상태를 확인한다.

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_name = 'USER'
  AND column_name IN ('status', 'withdrawn_at');
```

### 기대 결과

```text
All migrations have been successfully applied.
Database schema is up to date
The seed command has been executed.
```

`USER.status`, `USER.withdrawn_at` 컬럼이 조회되지 않는다.
운영 DB 확인은 schema 조회와 health check 같은 비파괴 검증으로 제한한다.

### 실패 시 조치

staging DB를 사용하는 경우 백업본으로 복구하고 migration 순서, 파괴적 변경 영향, Prisma schema와 실제 DB 차이를 확인한다.
staging DB를 운영하지 않는 경우 운영 배포를 중단하고 clean DB migration, 운영 read-only schema 조회, 백업 상태를 다시 확인한다.

### 증빙

|---|---|
| 검증 환경 | staging 미운영 대체 검증: clean DB + 운영 API/health 확인 |
| 실행 일시 | 2026-06-12 |

## OP-CHECK-006

### 항목

Rollback 절차 검증

### 수행 방법

`docs/12_deployment/롤백정의.md` 기준으로 staging 또는 운영 승인된 검증 환경에서 직전 안정 버전으로 되돌리는 절차를 수행한다.

staging DB/검증 서버를 별도로 운영하지 않는 경우에는 운영 배포 전 다음 대체 검증을 모두 수행한다.

1. 직전 안정 버전의 backend/frontend image 태그 또는 commit SHA를 확인한다.
2. 현재 배포 compose/env 파일에서 image 변수만 직전 안정 버전으로 되돌릴 수 있는지 확인한다.
3. 운영 DB backup 또는 복구 기준 시점을 확인한다.
4. rollback 실행 명령, 예상 영향, 재배포 복귀 명령을 운영 승인 기록에 남긴다.
5. 실제 운영 rollback은 장애 또는 승인된 rollback 상황에서만 수행한다.

```bash
docker images
docker compose down
docker compose up -d
```

롤백 후 health check를 확인한다.

```bash
curl -I https://kittysaemi.com/kittywallet/api/v1/health
curl -I https://kittysaemi.com/kittywallet/api/v1/health/db
```

Service Worker와 Cache Storage가 롤백 버전 기준으로 갱신되는지 확인한다.

```text
Chrome DevTools
 └ Application
     ├ Service Workers
     └ Cache Storage
```

### 기대 결과

* 직전 안정 버전 컨테이너가 기동된다.
* `/kittywallet/api/v1/health`와 `/kittywallet/api/v1/health/db`가 200으로 응답한다.
* PWA가 롤백 버전의 Service Worker를 활성화한다.
* Sync Queue와 IndexedDB 사용자 데이터는 삭제되지 않는다.
* staging 미운영 대체 검증에서는 직전 안정 image/commit, compose/env 복원 방법, DB backup 기준, 운영 승인 기록이 확인된다.

### 실패 시 조치

P1 장애로 분류하고 `docs/12_deployment/장애대응절차.md`와 `docs/12_deployment/백업복구정의.md`를 따른다.

### 증빙
