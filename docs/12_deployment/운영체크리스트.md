## OP-CHECK-004

### 항목

Nginx 설정 재생성 및 런타임 설정 검증

### 수행 방법

Nginx 설정 파일 변경이 포함된 배포에서는 컨테이너를 재생성한 뒤 설정 검증과 reload를 수행한다.

```bash
docker compose up -d --force-recreate nginx
docker compose exec -T nginx nginx -t
docker compose exec -T nginx nginx -s reload
```

실제 컨테이너가 새 라우팅 설정을 읽고 있는지 확인한다.

```bash
docker compose exec -T nginx nginx -T
```

### 기대 결과

```text
nginx: configuration file /etc/nginx/nginx.conf test is successful
```

`/kittywallet/api/` 라우팅과 `/kittywallet/` 프론트 라우팅이 런타임 설정에 포함된다.

### 실패 시 조치

Nginx 컨테이너를 재생성하고, `infra/nginx/default.conf` 마운트 경로와 배포 파일 업로드 상태를 확인한다.

### 결과

PASS / FAIL

## OP-CHECK-005

운영 배포 전 staging PostgreSQL 데이터베이스에서 현재 main 기준 migration을 적용한다.

```bash
npx prisma migrate deploy --schema apps/backend/prisma/schema.prisma
npx prisma migrate status --schema apps/backend/prisma/schema.prisma
```

회원탈퇴 물리 삭제와 관련된 컬럼 제거 상태를 확인한다.

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_name = 'USER'
  AND column_name IN ('status', 'withdrawn_at');
```

```text
All migrations have been successfully applied.
```

## OP-CHECK-006

### 항목

Rollback 절차 검증

### 수행 방법

`docs/12_deployment/롤백정의.md` 기준으로 staging 또는 운영 승인된 검증 환경에서 직전 안정 버전으로 되돌리는 절차를 수행한다.

```bash
docker images
docker compose down
docker compose up -d
```

롤백 후 health check를 확인한다.

```bash
curl -I https://kittysaemi.com/kittywallet/api/v1/health
curl -I https://kittysaemi.com/kittywallet/api/v1/health/db
```

Service Worker와 Cache Storage가 롤백 버전 기준으로 갱신되는지 확인한다.

```text
Chrome DevTools
 └ Application
     ├ Service Workers
     └ Cache Storage
```

### 기대 결과

* 직전 안정 버전 컨테이너가 기동된다.
* `/kittywallet/api/v1/health`와 `/kittywallet/api/v1/health/db`가 200으로 응답한다.
* PWA가 롤백 버전의 Service Worker를 활성화한다.
* Sync Queue와 IndexedDB 사용자 데이터는 삭제되지 않는다.

### 실패 시 조치

P1 장애로 분류하고 `docs/12_deployment/장애대응절차.md`와 `docs/12_deployment/백업복구정의.md`를 따른다.

### 증빙

| 실행자 |  |

## OP-API-003

### 항목

회원탈퇴 후 동일 이메일 재가입 검증

### 사전 조건

* staging 또는 운영 QA 계정을 사용한다.
* 테스트 계정의 미동기화 Sync Queue가 0건인지 확인한다.

### 검증 방법

1. QA 이메일로 회원가입한다.
2. 로그인 후 계좌, 카드, 카테고리, 거래를 1건 이상 생성한다.
3. 회원탈퇴 API 또는 화면에서 탈퇴를 수행한다.
4. 동일 이메일로 다시 회원가입한다.
5. 재가입 계정으로 로그인한다.
6. 탈퇴 전 생성한 데이터가 새 계정에 노출되지 않는지 확인한다.

### 기대 결과

* 탈퇴 요청이 성공한다.
* 동일 이메일 재가입이 성공한다.
* 재가입 계정에서 탈퇴 전 계좌, 카드, 카테고리, 거래가 조회되지 않는다.
* 서버 DB에 탈퇴 전 사용자 레코드가 남아 동일 이메일 재가입을 막지 않는다.

### 실패 시 조치

회원탈퇴 물리 삭제 transaction, `REFRESH_TOKEN` 폐기, 사용자 FK 삭제 순서를 확인한다.

### 증빙

|---|---|
| 테스트 이메일 |  |
| 탈퇴 일시 |  |
| 재가입 일시 |  |
| 기존 데이터 미노출 확인 |  |
| 실행자 |  |

## OP-API-004

### 항목

마이너스 통장 한도 검증

### 사전 조건

다음 테스트 계좌를 준비한다.

| 계좌 | 설정 |
|---|---|
| 마이너스 미허용 계좌 | `allow_negative_balance=false`, `negative_balance_limit=0` |
| 마이너스 허용 계좌 | `allow_negative_balance=true`, `negative_balance_limit` 0 초과 |

### 검증 방법

1. 마이너스 미허용 계좌에서 잔액을 초과하는 지출 거래를 등록한다.
2. 마이너스 허용 계좌에서 한도 내 지출 거래를 등록한다.
3. 마이너스 허용 계좌에서 한도를 초과하는 지출 거래를 등록한다.

### 기대 결과

| 검증 | 기대 결과 |
|---|---|
| 미허용 계좌 잔액 초과 지출 | 실패, `ACCOUNT_004` |
| 허용 계좌 한도 내 지출 | 성공 |
| 허용 계좌 한도 초과 지출 | 실패, `ACCOUNT_004` |

### 실패 시 조치

계좌 API 응답의 `allow_negative_balance`, `negative_balance_limit` 포함 여부와 거래 저장 시 날짜별 잔액 흐름 검증을 확인한다.
