# 3.1 릴리즈 생성

릴리즈는 `main` 브랜치 merge 후 GitHub Actions Release workflow와 semantic-release가 자동 생성한다.

Release workflow는 Node.js 22 LTS 기준으로 다음 검증을 모두 통과한 뒤에만 릴리즈를 생성한다.

```bash
npm ci
npm run lint
npm run type-check
npm run test
npm run build
```

semantic-release는 Conventional Commit을 기준으로 SemVer를 계산하고 다음 항목을 자동 갱신한다.

* `CHANGELOG.md`
* `package.json`
* `package-lock.json`
* Git tag
* GitHub Release

운영 배포 시 Docker image tag와 배포 기록은 생성된 Git tag를 기준으로 한다.
Deploy workflow는 GitHub Release가 published 상태로 생성된 뒤 실행되며, 생성된 Git tag 기준으로 Docker image를 build/push한다.

# 4. Docker Build

Docker build 단계의 Node 기반 build/runtime image는 Node.js 22 LTS 기준을 사용한다.

```bash
docker build -t ghcr.io/{owner}/kittywallet-api:{release-tag} .
docker build -t ghcr.io/{owner}/kittywallet-web:{release-tag} .
docker build -t ghcr.io/{owner}/kittywallet-ocr:{release-tag} -f apps/ocr-service/Dockerfile .
```

---

# 5. Docker Push

docker push ghcr.io/{owner}/kittywallet-api:{release-tag}
docker push ghcr.io/{owner}/kittywallet-web:{release-tag}
docker push ghcr.io/{owner}/kittywallet-ocr:{release-tag}

# 6. 서버 배포

```bash
docker compose pull
docker compose up -d
```

Nginx 설정 파일이 변경된 배포에서는 다음 명령으로 Nginx 컨테이너를 재생성한 뒤 설정을 검증한다.

```bash
docker compose up -d --force-recreate nginx
docker compose exec -T nginx nginx -t
docker compose exec -T nginx nginx -s reload
```

`infra/nginx/default.conf`는 bind mount로 연결되므로 파일 교체가 발생한 배포에서 reload만 수행하면 컨테이너가 이전 설정을 계속 사용할 수 있다.

# 7. DB Migration 및 Seed

```bash
npx prisma migrate deploy
npm run db:seed
```

Migration 완료 후 기본 아이콘/카테고리 같은 공통 기준 데이터는 seed로 적재한다.
Seed는 중복 실행 가능해야 하며, 기존 데이터를 삭제하거나 사용자 데이터를 변경하지 않는다.
